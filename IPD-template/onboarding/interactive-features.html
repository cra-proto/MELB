How interactive features work

The site uses three JavaScript-powered features beyond the standard WET-BOEW toolkit.

On this page

  • Copy-to-clipboard
  • Interactive questions (fieldflow)
  • components.js helpers

Copy-to-clipboard (clipboard.js)

A 463-line WET-BOEW plugin written by Andrew de Peiza. It follows the WET plugin architecture (wb.init, wb.add, timerpoke pattern).

What it does: Automatically generates a “Copy” button next to any element with the class wb-clipboard. When clicked, it copies the element’s content to the clipboard and shows a green “Copied” feedback state.

Key features:

  • Multi-method copy: tries document.execCommand("Copy") first, falls back to the async Clipboard API
  • Bilingual: built-in English (“Copy”/“Copied”) and French (“Copier”/“Copie”) strings
  • Accessible: adds screen-reader text around copy areas, uses aria-live="polite"
  • Supports contenteditable elements with change detection

Used on: About 8 component pages that show code examples (alerts, tables, and-or, interactive-questions, etc.).

To add to a page: Include clipboard.css and clipboard.js in the page’s <head> and footer, then add the wb-clipboard class to any element whose content should be copyable.

Interactive questions (fieldflow)

WET-BOEW’s wb-fieldflow component renders decision trees as radio buttons or dropdowns. It is used on component pages for design option selection and on the contact-us page for routing users to the right phone number.

The configuration is passed via data-wb-fieldflow JSON attributes. Key options:

  • "renderas": "radio" — renders as radio buttons (recommended)
  • "gcChckbxrdio": true — uses GC-styled checkboxes/radios
  • "noForm": true — prevents form submission behaviour
  • Nested questions use the wb-fieldflow-sub class

components.js helpers

A tiny 13-line file loaded on about 20 pages. It does two things:

  1. Auto-selects the first radio button in .wb-fieldflow components (sets a default selection)
  2. Handles data-inview events from WET’s wb-inview plugin, updating .view-state-status elements when content scrolls into view
