The universal COP lifecycle

Every COP followed the same macro pattern, though with varying levels of formality:

  1. Baseline/Scaffolding – Import current Canada.ca pages, set up repo structure
  2. Intensive Build Sprint – IA restructuring, iterative prototyping, co-design sessions
  3. SME Review – Subject matter experts review content for accuracy
  4. Validation Testing – Task-based usability testing with real participants
  5. Refinement – Incorporate test findings, polish content
  6. Long-tail Maintenance – Ongoing updates post-launch

The GST/HST Business COP documented this most explicitly in its ROADMAP.md with a Gantt chart, but the pattern is visible in commit history across all projects.

Phase 1: How a COP starts

Who starts it: A technical or IA lead creates the repository, sets up templates, and builds the initial page skeleton.

What the first days look like across different COPs:

  • EOL COP: 11 commits on day one – "Basic page set" with 6 pages (index, represent-someone-died, list-assets, manage-estate-debt, prepare-returns, types-deadlines)
  • TFSA COP: 97 commits in the first 2 days – IA/Tech Lead 2 created the entire IA structure (15+ pages) in a single burst
  • GST/HST COP: 30 commits of README/infrastructure first, then baseline content import over the next 3 weeks
  • CAP COP: Platform Developer 1 uploaded initial files – index, send-payment, payment-arrangements, find-payment, options pages

The baseline typically captures the current Canada.ca page structure before any optimization begins. The GST/HST project explicitly imported scoped files from the live site (Oct 8–18, 2024), giving them a "before" snapshot to work from.

What happens during the sprint:

IA restructuring

Pages get split, merged, renamed, and reorganized. The EOL project split its clearance certificate from 1 page into 9 sub-pages. The GST/HST project deleted 36 redundant pages and consolidated filing methods from siloed sections into a task-based hub.

Iterative prototyping via numbered file variants

Instead of git branches, teams create _proto1, _proto2, _proto3 etc. of key pages. The GST/HST "choose your filing method" page went through 7+ named versions. Business Registration had 12 variants of how-register-bn. TFSA had 4 variants of the topic page and 4 variants of the excess contributions page.

Co-design sessions feed directly into commits

GST/HST commit messages reference "Updates from codesign 1", "Text updates from COP meeting", "Feb 4 COP meeting updates", "Friday meeting edits" – showing regular collaborative sessions driving changes.

Content hiding rather than deleting

A distinctive pattern across all COPs – old content is commented out or hidden rather than removed, allowing side-by-side comparison. TFSA commits show "Hid H2 section", "Hid old content", "Hid mermaid flow chart". This lets the team compare old vs. new approaches.

Team roles and structure

The typical COP team is 3–4 people with one technical/IA person and 1–2 content designers, plus external SME reviewers whose feedback gets incorporated through named commits.

The development workflow

No COP uses git branching for content versioning. Every single repo has only main (plus occasional stale patch branches from GitHub's web editor). Instead:

  • File-level versioning: Numbered variants (_proto0 through _proto6, or -v2, -v3, or -2, -3)
  • Archive directories: Old versions moved to /old/ or /archive/ folders
  • Parallel repos instead of branches: EOL created EOL-2 as a separate testing copy. TFSA created separate test repos as forks. GST/HST uses a dual-org fork pair.
  • Direct commits to main via GitHub's web editor: The dominant workflow across all COPs. Content designers edit HTML directly in the browser.

The org relationship

  • cra-proto is the prototyping org – repos are created here first, and GitHub Pages serves the prototypes
  • cra-design is the design workspace – some COPs fork here for active daily work (GST/HST, TFSA)
  • proto-cra appears to be an older/legacy org with less COP activity

The migration pattern is: start in cra-design → fork/migrate to cra-proto when the prototype needs to be "official". The TFSA COP explicitly shows this with commit messages referencing the repo migration.
