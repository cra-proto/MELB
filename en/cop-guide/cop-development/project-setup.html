Project setup standards – How a COP is developed

The GST/HST Business COP is the most documented project and serves as the template a junior designer should follow. This section covers the exact repository structure, configuration, and documentation standards used.

Repository file structure

A well-structured COP repo follows this pattern:

Standard COP repository directory layout
Path Purpose
_config.ymlJekyll configuration (site-wide settings)
README.mdProject overview, links, roadmap checklist
index.htmlScoped page inventory (the master list of all pages in scope)
exit-intent.htmlExit page for links leaving the prototype
404.htmlCustom error page
sitemap.htmlSite map page
docs/CONTRIBUTING.mdProto naming rules, update procedures
docs/ROADMAP.mdPhase checklist and Gantt chart
docs/CHANGELOG.mdAdded/Deleted/Modified page tracker

templates/Page templates (Jekyll and CDTS versions), CSS, scripts
en/All content pages (mirrors Canada.ca URL paths)
old/Archived prototype versions

Jekyll configuration

Every COP repo uses the gcweb-jekyll remote theme. This is what a standard _config.yml looks like:

Example _config.yml 
baseurl: /your-cop-name
remote_theme: wet-boew/gcweb-jekyll
title: "Your COP Title"
description: "Project description"
developerOptions: true
exitPage: true
testBanner: true

Key settings:

  • baseurl – Must match the repository name for GitHub Pages to work
  • remote_theme – Always wet-boew/gcweb-jekyll for GCWeb/WET styling
  • developerOptions – Set to true during development, must be set to false before user testing
  • exitPage – Enables the exit intent page for external links
  • testBanner – Shows a banner indicating this is a prototype, not the live site

Contributing rules

From the GST/HST COP's CONTRIBUTING.md, these are the rules every team member must follow:

  1. Proto naming: Prototype files get _proto# appended → choose-filing-method_proto1.html
  2. Branch + PR workflow: Create a new branch, then pull request to merge (though in practice many teams commit directly to main)
  3. PR/commit naming: Must identify what was updated or changed
  4. Path structure changes: Must be communicated to all repo contributors
  5. Absolute links only: No relative links – all links must be absolute URLs
  6. File naming = final Canada.ca URL: Create files where they'll live on Canada.ca, following GC URL guidelines
  7. Developer options off for testing: Set developerOptions: false in _config.yml before user testing

Proto naming convention

This is one of the most important COP conventions. Instead of using git branches for versioning, COPs use file-level versioning through numbered suffixes:

Proto naming system explained
Suffix Meaning Example
_proto0 Baseline – the current live Canada.ca page as-is how-file_proto0.html
_proto1 First proposed redesign how-file_proto1.html
_proto2, _proto3, etc. Subsequent iterations how-file_proto2.html
(no suffix) The current working version / final version how-file.html

When a prototype version is no longer active, it gets moved to the /old/ directory. This keeps the working directory clean while preserving design history.

Key insight: The _proto0 file is your “before” snapshot. Keeping it lets the team compare old vs. new approaches throughout the project, and it becomes critical evidence when measuring improvements during validation testing.

Required documentation

A mature COP repo includes these documentation files in the docs/ directory:

ROADMAP.md – Phase checklist and timeline

Tracks which COP phases are complete. The GST/HST COP used a Gantt chart showing actual durations:

  • Create repo + templates: 3 days
  • Baseline user testing: 19 days
  • Import scoped files: 9 days (overlaps with testing)
  • Evaluate baseline testing: 14 days
  • Content and design prototyping: 134 days
  • Validation testing: 20 days
CHANGELOG.md – The production handoff deliverable

Three tables tracking every page action with exact URLs. This is the deliverable the web publishing team uses to implement changes on Canada.ca:

Table What it tracks
Pages to be addedNew pages with GitHub URL + new Canada.ca URL
Pages to be deletedPages to remove, with current Canada.ca URL + redirect target
Pages to be modifiedChanged pages with old title, old URL, new title, new URL, and whether the modified date needs updating
CONTRIBUTING.md – Team workflow rules

Proto naming conventions, commit message standards, link format rules, and file naming guidelines as described above.

SUPPORT.md – Technical reference

Jekyll setup guide, YAML front matter reference, GCWeb template options, and troubleshooting tips for team members who are new to the tooling.

YAML front matter reference

Every HTML page in a COP needs YAML front matter at the top. Here is the standard format used in subway child pages:

Complete YAML front matter example 
---
layout: without-h1
title: "Page Title – Parent Section Title"
description: "Brief page description"
subject: "Economics and Industry;Taxes"
keywords: "keyword1, keyword2"
auth:
  type: "contextual"
  label: "Sign in"
  labelExtended: "CRA sign in"
  link: "https://www.canada.ca/en/revenue-agency/services/e-services/cra-login-services.html"
altLangPage: "/fr/equivalent-page.html"
dateModified: 2025-05-13
dateIssued: 2023-08-07
breadcrumbs:
  - title: "Taxes"
    link: "https://www.canada.ca/en/services/taxes.html"
  - title: "Parent Section"
    link: "https://example.com/parent"
feedbackData:
  section: "Page Title"
---

Key fields:

  • layout: without-h1 – Required for subway child pages (the H1 is rendered in the page body after the subway nav)
  • layout: default – Used for non-subway pages and landing pages
  • auth – Adds a contextual "Sign in" button to the page
  • breadcrumbs – Defines the breadcrumb trail (must use absolute URLs)
  • feedbackData – Configures the page feedback widget
