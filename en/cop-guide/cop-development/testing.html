Testing methodology – How a COP is developed

Every COP follows a two-phase testing approach: baseline testing on the current Canada.ca pages (before any design work), followed by validation testing on the redesigned prototype. The GST/HST Business COP has the most complete testing infrastructure and serves as the model.

Phase 1: Baseline testing

Baseline testing happens before any design work begins. The team tests the existing Canada.ca pages to understand where users struggle.

From the GST/HST ROADMAP Gantt chart:

  • Baseline user testing and notetaking: Sep 26 – Oct 14, 2024 (19 days)
  • Evaluation of baseline testing: Oct 15 – Oct 28, 2024 (14 days)

The import of scoped files to the repo (Oct 10–18) overlaps with testing – the team is simultaneously setting up the baseline prototype while still testing. This overlap is intentional and efficient.

Why baseline testing matters: Without a baseline, you cannot measure whether your redesign actually improved anything. The baseline gives you concrete task completion rates to compare against validation results.

Phase 2: Validation testing

Validation testing happens after the prototype is largely complete. The same types of task-based scenarios are run against the redesigned prototype to measure improvement.

In the GST/HST COP, validation testing infrastructure was built in July 2025 – 10 months into the project. This is typical: testing is a late-stage activity, not a design input.

Test infrastructure structure

The testing infrastructure lives in a dedicated directory within the COP repo:

File Purpose
instructions.htmlExplains the test process to participants

task-1.html through task-6.htmlIndividual task scenario pages
post-task-survey.htmlFollow-up questions after each task

How a test session works

Based on the GST/HST testing infrastructure:

  1. A moderator presents tasks one at a time
  2. The participant reads the task scenario on the task page
  3. The participant clicks "Start task" which opens the prototype in a new browser tab
  4. The participant uses the prototype to try to complete the task
  5. The participant comes back to the task tab and selects their answer from multiple-choice options
  6. The participant answers two follow-up questions about the experience
  7. Throughout, the participant is asked to think aloud as they navigate

This approach separates the task prompt (on the task page) from the prototype (in a new tab), so the participant's navigation behaviour in the prototype is natural and unguided.

Writing effective task scenarios

Tasks are presented as realistic scenarios with specific, verifiable answers. Here is an example pattern from the GST/HST COP:

Example task scenario:

"You need to file your GST/HST return, but you don't have a GST/HST access code for filing. Your company is registered with a CRA account and you want to try filing through there. Do you need an access code to file your return through your CRA account?"

Notice the pattern:

  • Sets a realistic context – the participant has a specific situation
  • Has a specific, verifiable question – there is one correct answer
  • Tests navigation ability – the participant must find the right page and the right section to answer
  • Multiple-choice answers – allows quantitative measurement of task success

Analytics and session tracking

Test pages include two forms of analytics tracking:

Tool What it captures Why it matters
Google Analytics Page views, navigation paths, time on page Shows which pages participants visited and how long they spent
Microsoft Clarity Session recordings, heatmaps, scroll depth Lets the team watch actual participant behaviour, not just task completion rates

Both are added via script tags in the test instruction pages. The analytics tracking is only on the testing infrastructure pages, not on the prototype content pages themselves.

Preparing the prototype for testing

Before running any user testing, you must prepare the prototype:

  1. Set developerOptions: false in _config.yml – this hides editing buttons and GitHub links so participants see a clean Canada.ca-looking page
  2. Ensure all links work – broken links during testing invalidate results
  3. Use absolute links throughout – relative links may break when the prototype is accessed via GitHub Pages
  4. Test the test – run through each task scenario yourself to verify the correct answer is findable

Critical: The developerOptions: false setting is mandatory before testing. If participants see developer tooling, it breaks the illusion that they are navigating a real Canada.ca page, and their behaviour will not be representative.

When testing happens in the COP lifecycle

Testing is bookended around the design work:

Testing timeline from the GST/HST COP
Phase When Duration
Baseline user testing Before any content changes (Sep – Oct 2024) 19 days
Evaluate baseline results Immediately after baseline testing 14 days
Content and design prototyping After baseline evaluation 134 days
Validation testing After prototyping is substantially complete (Jul 2025) 20 days

Across all COPs, accessibility testing and usability testing are late-stage validation activities. Testing validates the design – it does not drive it. Co-design sessions and SME review drive the design during the build sprint.
